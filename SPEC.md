# CapDrop（截落）— 智能截图工具

> 版本: 1.1 | 更新: 2026-04-18

## 1. 项目概述

- **项目名称**：CapDrop（中文名：截落）
- **一句话描述**：截图即保存——截屏、标注、自动存入目标位置，三步完成
- **目标用户**：学习笔记党、程序员、内容创作者（频繁截图并粘贴到文档的人）
- **核心痛点**：现有截图工具（微信/Snipaste/系统自带）截图后需要手动保存/粘贴，步骤繁琐；选区不精准；编辑标注粗糙
- **项目话题**：Telegram #capdrop
- **Discord 频道**：Discord #capdrop

## 2. 需求梳理（按优先级）

### 2.1 核心需求（P0 — MVP 必须）

**1. 快捷键触发截屏**
- 全局快捷键，一键启动截图（避免与系统/主流软件冲突）
- 备选方案：Ctrl+Alt+` 或 F7~F12 系列
- 首次使用时弹窗提示设置快捷键，可自定义

**2. 智能选区**
- 鼠标移动时，自动感知并高亮窗口/UI元素边界（吸附效果）
- 直接点击 = 自动截取该元素
- 拖拽 = 严格按鼠标起止位置确定区域
- 可选：固定区域模式（截屏区域不变，只换内容）

**3. 轻量标注编辑器**
- 截图后立即进入编辑模式（覆盖层，非弹窗）
- 工具栏：画笔、箭头、矩形框、圆形、文字、马赛克
- 工具栏样式：简约图标，图标大小固定（不受截图尺寸影响）
- 画笔：曲线图标表示，支持粗细和颜色选择
- 箭头：方向与直线一致
- 马赛克：涂抹即见效果，不可调整大小
- 撤销/重做
- 可选：截图注释文字框（输入后自动追加到 Markdown 图片下方）

**4. 自动保存到预设位置**
- 截图确认后，自动保存到用户预设的目标位置
- 支持的目标类型（可同时选多个）：
  - **剪贴板**（默认必选，无需在界面显示）
  - **本地文件夹**（自动备份，可选）
  - **Markdown 文档**（图片 + 相对路径引用，自动追加到文档末尾）
  - **桌面应用窗口**（选择已打开的窗口，截图直接粘贴进去）
- 保存目标配置界面：简洁列表，每行一个目标 + 路径/配置
- 命名规则：可配置（如时间戳、序号等）

**5. 本地图片自动备份**
- 每次截图自动保存一份到本地默认文件夹
- 避免截图丢失

### 2.2 次要需求（P1 — 后续迭代）

- OCR 文字识别
- 翻译功能
- 飞书 Webhook 集成（通过群机器人发送图片）
- 滚动截图
- 历史截图管理
- 置顶桌面（贴图）

### 2.3 明确不做（防止范围蔓延）

- 不做云存储/上传服务
- 不做录屏/GIF
- 不做账号系统
- 不做付费功能（初期免费）
- 不做 API 鉴权集成（飞书等只用 Webhook）

## 3. 非功能性需求

### 3.1 轻量化（核心约束）

- **安装包 < 20MB**（单文件可执行，免安装）
- **内存占用 < 100MB**
- **启动时间 < 2秒**
- 无外部运行时依赖（不要求用户安装 .NET/Python 等）

### 3.2 平台支持

| 优先级 | 平台 | 备注 |
|--------|------|------|
| P0 | Windows 10/11 x64 | 主要目标，用户主力机 |
| P1 | macOS 12+ | 开发机验证 |
| P2 | Linux | 可选 |

### 3.3 色彩保真度

- 截图色彩必须与屏幕一致，不能出现"发灰"、"降灰度"现象
- 正确处理色彩空间（sRGB）、Gamma值、DPI缩放
- 保存为 PNG 无损格式

## 4. 技术方案

### 4.1 推荐方案：Tauri 2（Rust + WebView）

**为什么选 Tauri 而不是之前的方案：**

| 方案 | 问题 |
|------|------|
| Python + PyQt5 | 打包 270MB→33MB（优化后），依赖多，启动慢 |
| .NET WPF | 需安装运行时，非自包含版用户需额外下载 |
| Electron | 打包 > 100MB，内存占用高 |
| **Tauri 2** | **打包 3-10MB，Rust 原生性能，内置截图 API，无运行时依赖** |

**架构：Hexagonal Architecture（六边形架构）**
- Core：截图逻辑、保存规则、快捷键管理（Rust）
- Ports：截图引擎接口、存储接口、剪贴板接口
- Adapters：平台特定实现（Windows/macOS 截图API）
- UI：WebView（HTML/CSS/JS），轻量前端框架

### 4.2 技术栈

- **核心**：Rust（Tauri 2）
- **UI**：HTML + CSS + TypeScript（轻量框架，如 Preact 或 Vanilla）
- **截图引擎**：Windows 用 windows-rs (WGC/DXGI)，macOS 用 core-graphics
- **配置存储**：JSON 文件（本地，无数据库）
- **构建**：GitHub Actions CI/CD → 自动构建 Windows .exe / macOS .dmg

## 6. 失败经验总结（来自之前尝试）

> 以下是从文档中提炼的"踩坑记录"，新方案必须规避：

| 问题 | 原因 | 新方案对策 |
|------|------|-----------|
| Codex/Claude 频繁断线、积分耗尽 | AI 编程工具不稳定、额度管理差 | Hermes 多模型降级策略，Claude Code + GLM 5.1 双保险 |
| 快捷键冲突（Alt+Q 无效） | 与系统/其他软件冲突 | 首次启动检测冲突，引导用户自定义 |
| 截图"发灰" | 色彩空间/Gamma/DPI 处理不当 | Rust 原生 API 直接获取正确色彩数据 |
| 打包体积过大 / 需安装运行时 | Python/.NET 依赖 | Tauri 2 单文件打包，无运行时依赖 |
| 每次修改都需要重新编译 30min+ | Rust 首次编译慢 | 增量编译 + CI/CD 自动构建 |
| AI 生成的代码前后不一致 | 上下文丢失，提示词不精准 | SPEC.md + CLAUDE.md 作为持久上下文 |
| 选区不精确 | 简单矩形框选 | 智能吸附 + 窗口元素检测 |

## 8. 风险 & 待决

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| Tauri 2 截图 API 在 Windows 上不够成熟 | 需要自己写 Rust 调用 Win32 API | 中 | 备选：用 windows-rs 直接调用 WGC |
| Mac Mini 无 Rust 环境 | 阻塞开发启动 | 低 | `curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs \| sh` |
| 前端标注编辑器复杂度高 | UI 开发周期拉长 | 中 | 先用 Canvas API 做基础版，后续优化 |
| 跨平台色彩处理差异 | Windows/macOS 截图效果不一致 | 中 | 各平台独立适配层 |

**待决问题：**
- 正式命名：CapDrop vs AutoCap vs 智存截图（先开发，上线前定）
- Windows 构建方案：本地 VM vs GitHub Actions
